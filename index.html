Vulnerability je isključen

1.SQL injection

Ako bi unijeli u donji input: "1" dobili bi usera sa id=1, dok je vulnerability uključen moguće je napisati: "1 or (1=1)" koji bi vratio sve usere iz tablice user (Ilegalni upiti)

Također ako bi unijeli: "1; select * from info" ne samo da bi dobili informacije o korisniku sa id 1 također bi dobili sve podatke u tablici info (Ulančani upiti)

Kada isključimo vulnerability koristimo query placeholder-e od mysql library-a koji onemogućuju ovakve napade.

Unesite id user-a:

  • {{user}}

2. Cross-site scripting (XSS)

Dok je vulnerability uključen moguće je u mjesto za poruku unijeti: <a onmouseover=window.location.replace("http://www.w3schools.com")>click me! </a>

Kada bi misom dodirnuli "click me!" preusmjerio bi nas na drugu stranicu (potencijalno na stranicu napadača.)

Također ako bi stavili u poruku <img src="notValidUrl" onerror=alert('HACKED!')> bi se u browseru prikazao alert HACKED! (da ova aplikacija nema auth0 autorizaciju te da su cookies postavljeni na "HttpOnly: false" u alert bi mogli staviti document.cookie te bi nam alert prikazao cookie login sjednice te bi napadač mogao preuzeti sjednicu)

Kada isključimo vulnerability sav tekst napisan u poruci se automatski sprema kao string te prijašnje ranjivosti neće biti moguće.

Unesi poruku:

{{message}}

3. Cross Site Request Forgery (CSRF)

Dok je vulnerability uključen svaki put kada se klikne gumb: "Pošalji poruku" radi se o GET zahtjevu koji pošalje napisanu poruku. Kada kliknemo na gumb: "Napadačeva stranica" preusmjerava nas na napadačevu stranicu gdje bi se trebala učitati slika no u src dijelu slike se nalazi GET zahtjev za poruku HACKED!. (ta poruka se pošalje automatski čim browser korisnika automatski izvodi GET zahtjev unutar src djela img atributa, također korisnik odlaskom na napadačevu stranicu i dalje ima cookie od originalne stranice ako se nije odlogirao te zbog toga može napraviti takav zahtjev.)

Nakon što otvorite napadačevu stranicu potrebno je ponovno učitati ovu stranicu da se vidi da je napadač poslao poruku HACKED!. Naravno nakon što se isključi vulnerability to više neće biti moguće jer se zahtjev pretvara u POST te više nije moguće preko napadačeve stranice poslati poruku. (također da smo omogućili CORS i to bi bilo rješenje)

Napadačeva stranica
